# 第 4 部分 DevOps 与服务质量评价

## 4.1 DevOps 技术运用

本项目在开发与部署过程中使用了 GitHub、Docker、Jenkins、Kubernetes 等 DevOps 工具，形成了从代码提交、自动测试、镜像构建到自动部署的完整流水线。代码统一托管在 GitHub 仓库 `guide-agent` 中，开发完成后通过 `git push` 将代码提交到远程仓库。Jenkins Pipeline 通过 `Pipeline script from SCM` 从 GitHub 拉取最新代码，并读取项目根目录下的 `Jenkinsfile` 执行自动化流程。

Jenkins 流水线主要包含以下阶段：代码拉取、环境检查、创建 Python 虚拟环境、安装依赖、静态编译检查、执行 `evaluate.py` 自动化测试、构建 Docker 镜像、推送镜像仓库、部署到 Kubernetes、执行 Smoke Test。流水线通过参数 `DEPLOY_TARGET` 控制部署目标：当参数为 `none` 时只执行测试和镜像构建；当参数为 `compose` 时部署到 Docker Compose；当参数为 `k8s` 时推送镜像并部署到 Kubernetes 集群。

项目使用 Dockerfile 将 Agent Web 服务、业务代码、工具集和前端页面封装为统一镜像。Kubernetes 部署文件位于 `k8s/` 目录，包含 `agent-app`、`order-service`、`product-service`、`logistics-service` 四类服务，以及 ConfigMap 和 Secret 配置。真实 LLM API Key 不写入代码仓库，而是通过 Kubernetes Secret `service-agent-llm-secret` 注入运行环境，从而保证部署安全性。

## 4.2 服务监控

本项目使用自动化监控工具采集服务运行参数。监控工具包括 Prometheus、Grafana、cAdvisor、项目自带脚本 `scripts/quality_monitor.py`、Docker 命令、Kubernetes 命令和 Jenkins 控制台日志。其中 Prometheus 用于周期性抓取服务指标，Grafana 用于可视化展示，cAdvisor 用于采集容器 CPU、内存、网络等资源消耗指标，`quality_monitor.py` 用于主动产生业务请求并生成服务质量评价数据。

自动化监控工具启动命令如下：

```powershell
docker compose -f docker-compose.yml -f docker-compose.monitoring.yml up -d --build
```

监控界面访问地址如下：

```text
Prometheus: http://127.0.0.1:9090
Grafana:    http://127.0.0.1:3000  用户名/密码 admin/admin
cAdvisor:  http://127.0.0.1:8081
```

Prometheus 自动抓取 `agent-app` 暴露的 `/metrics` 指标，主要包括 HTTP 请求总数、对话请求意图分布、对话接口延迟直方图等。Grafana 已预置 `Guide Agent 服务质量监控` 仪表盘，展示 HTTP 吞吐率、P50/P95 延迟、各意图调用量、接口成功率、容器 CPU 使用和容器内存使用。cAdvisor 提供容器级资源监控页面，可用于查看各容器 CPU、内存、网络 I/O 等消耗情况。

项目自带监控脚本运行命令如下：

```powershell
.\.venv\Scripts\python.exe scripts\quality_monitor.py --base-url http://127.0.0.1:8000 --rounds 5 --concurrency 2 --timeout 90
```

该脚本会向 Web Agent 的 `/api/chat` 接口发送订单、物流、售后、导购等多类请求，用于产生真实业务调用流量，同时采集调用参数和时间参数，包括总请求数、成功请求数、失败请求数、状态码分布、平均延迟、P50 延迟、P95 延迟、最大延迟和吞吐率。同时脚本会尝试读取 Docker 与 Kubernetes 的资源状态，包括 `docker compose ps`、`docker stats --no-stream`、`kubectl get pods`、`kubectl get svc`、`kubectl get deploy`、`kubectl top pods` 等命令输出。

本次监控数据保存在：

```text

reports/quality/20260707_143230/summary.md
```

## 4.3 服务质量评价

本项目包含的主要软件服务包括：Web Agent 服务 `agent-app`、订单微服务 `order-service`、商品微服务 `product-service`、物流微服务 `logistics-service`，以及 Agent 内部调用的导购工具集和售后 BPMN 流程。系统主要 API 接口包括 Web 页面入口 `/` 和对话接口 `/api/chat`，其中 `/api/chat` 会根据用户意图路由到订单、物流、商品咨询、售后 BPMN 或增强导购 Agent。

### 4.3.1 效率评价

效率主要通过响应时间评价，监控指标包括平均延迟、P50 延迟、P95 延迟和最大延迟。本次监控中平均延迟为 19230.56 ms，P95 延迟为 30008.52 ms，最大延迟为 30009.52 ms。该结果说明在接入真实 LLM 后，导购和部分路由请求的耗时明显增加，整体效率仍有优化空间。

延迟较高的主要原因是：导购增强路径会先收集商品候选、评价摘要、销量信号和店铺售后信息，再调用真实 LLM 基于 evidence 证据包进行综合推理。真实 LLM 网络请求本身存在不稳定性和较长响应时间，因此比普通规则查询和 BPMN 流程更慢。

优化方向包括：对普通商品咨询与购买决策问题进一步区分，避免简单价格查询误入导购推理；对商品 evidence 证据包增加缓存；对真实 LLM 设置合理超时和 fallback；对导购推理结果进行短期缓存；在生产环境中使用异步任务或流式响应降低用户感知等待时间。

### 4.3.2 可用性评价

可用性采用成功率进行评价：

```text
可用性 = 成功请求数 / 总请求数
```

本次监控中总请求数为 20，成功请求数为 16，失败请求数为 4，成功率为 80.00%。失败请求的错误类型为 `timed out`，主要出现在真实 LLM 导购请求中。这说明系统的基础服务部署可用，Kubernetes 中各 Pod 均处于 Running 状态，但真实 LLM 调用链路会受到网络和模型响应时间影响。

因此，本次可用性评价为：基础微服务和 K8s 部署可用性较好，真实 LLM 导购链路在 30 秒超时条件下可用性不足。项目已通过延长 smoke test 和质量监控超时时间、保留规则推荐 fallback 的方式进行改进，以保证真实模型不可用或超时时系统仍能返回可用推荐结果。

### 4.3.3 健壮性评价

健壮性主要从异常处理、服务重启、fallback 能力和测试保障四个方面评价。

系统具备以下健壮性设计：第一，若没有配置 `OPENAI_API_KEY`，`llm.py` 会自动使用 MockLLM；第二，导购 Agent 在真实 LLM 不可用、调用失败或 evidence 为空时，会回退到 `tools.recommend_products` 规则推荐；第三，Kubernetes Deployment 管理 Pod，当容器异常退出时可自动重启；第四，Jenkins 在部署前执行 `py_compile` 和 `evaluate.py`，部署后执行 smoke test，降低错误版本进入运行环境的概率。

从本次监控结果看，系统发生了 4 次超时失败，说明真实 LLM 调用链路仍需优化。但订单、物流、售后和部分商品/导购请求能够成功返回，Kubernetes 服务也保持 Running 状态，因此系统具备一定健壮性。后续可通过更细粒度的超时控制、缓存、重试、熔断和降级策略进一步增强健壮性。

### 4.3.4 吞吐率评价

吞吐率计算公式为：

```text
吞吐率 = 总请求数 / 总耗时
```

本次监控总请求数为 20，总耗时约 203.8596 秒，吞吐率为 0.0981 req/s。吞吐率偏低，主要原因是真实 LLM 请求耗时较长，并且监控脚本设置了并发数 2，用于课程演示而非高并发压测。

课程实验场景下，该吞吐率可以证明系统能够完成多类业务请求的端到端处理；若面向真实生产环境，需要进一步使用 JMeter、Locust 或 K6 进行更大规模压测，并结合多副本扩容、缓存、异步化和模型网关优化提升吞吐能力。

### 4.3.5 综合评价

综合来看，本项目已经形成从开发、测试、构建、部署到监控评价的闭环。DevOps 方面，项目使用 GitHub 管理代码，使用 Jenkins 自动执行测试、构建镜像和部署，使用 Docker/Kubernetes 实现服务容器化和编排，使用 K8s Secret 管理真实 LLM 配置。服务质量方面，订单、商品、物流、售后 BPMN 和导购 Agent 均可通过 Web 统一入口访问，Kubernetes Pod 和 Deployment 状态正常。

本次监控也暴露了真实 LLM 场景下的性能问题：响应时间较长、部分导购请求超时、吞吐率偏低。因此，当前系统适合作为课程实验和功能展示系统，具备较好的可演示性、可部署性和基础健壮性；若进一步面向生产环境，需要重点优化真实 LLM 调用耗时、缓存策略、fallback 策略和并发处理能力。
