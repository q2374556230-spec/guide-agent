# 第 4 部分 DevOps 与服务质量评价

## 4.1 DevOps 技术运用

本项目在开发、测试、构建和部署过程中使用了 GitHub、Jenkins、Docker、Docker Compose、Kubernetes、Prometheus、Grafana、cAdvisor 等 DevOps 与监控工具，形成了从代码提交到服务运行监控的完整闭环。

代码托管在 GitHub 仓库 `guide-agent` 中。开发完成后，代码通过 Git 提交到远程仓库；Jenkins 使用 `Pipeline script from SCM` 从 GitHub 拉取最新代码，并读取项目根目录下的 `Jenkinsfile` 执行自动化流水线。流水线主要阶段包括：代码拉取、环境检查、创建 Python 虚拟环境、安装依赖、静态编译检查、执行 `evaluate.py` 自动化测试、构建 Docker 镜像、推送镜像仓库、部署到 Kubernetes、执行 Smoke Test。

Jenkins 通过参数 `DEPLOY_TARGET` 控制部署目标：当参数为 `none` 时只执行测试和镜像构建；当参数为 `compose` 时部署到 Docker Compose；当参数为 `k8s` 时推送镜像并部署到 Kubernetes 集群。当前项目已完成 Windows Jenkins 适配，流水线中使用 `bat` 和 `powershell` 执行命令，并在 workspace 中创建 `.venv` 虚拟环境，避免依赖系统 PATH 中不确定的 Python。

项目使用 Dockerfile 将 Web Agent、导购 Agent、业务工具、BPMN 流程和前端页面打包为统一镜像。Docker Compose 用于本地容器化部署，Kubernetes 用于服务编排。K8s 部署资源位于 `k8s/` 目录，包含 `agent-app`、`order-service`、`product-service`、`logistics-service` 四类服务，以及 ConfigMap 和 Secret 配置。真实 LLM API Key 不写入代码仓库，而是通过 Kubernetes Secret `service-agent-llm-secret` 注入运行环境，从而降低敏感信息泄露风险。

## 4.2 服务监控

本项目采用自动化监控工具采集服务运行参数。监控工具包括 Prometheus、Grafana、cAdvisor 和项目自带的 `scripts/quality_monitor.py`。其中，Prometheus 周期性抓取服务暴露的 `/metrics` 指标，Grafana 负责可视化展示监控数据，cAdvisor 负责采集容器 CPU、内存、网络等资源消耗指标，`quality_monitor.py` 用于主动产生订单、物流、售后和导购请求，并生成服务质量评价数据。

自动化监控工具启动命令如下：

```powershell
docker compose -f docker-compose.yml -f docker-compose.monitoring.yml up -d --build
```

监控界面访问地址如下：

```text
Prometheus: http://127.0.0.1:9090
Grafana:    http://127.0.0.1:3000  用户名/密码 admin/admin
cAdvisor:  http://127.0.0.1:8081
```

Prometheus Targets 页面显示当前监控目标均已正常接入：

| 监控目标 | 状态 | 说明 |
| --- | --- | --- |
| `cadvisor` | UP | 采集容器 CPU、内存、网络等资源数据 |
| `guide-agent-app-compose` | UP | 采集 Docker Compose 中 `app:8000/metrics` 指标 |

Agent Web 服务在 `/metrics` 暴露 Prometheus 格式指标，主要包括：

## 4.3 服务质量评价

本项目包含的主要软件服务包括：Web Agent 服务 `agent-app`、订单微服务 `order-service`、商品微服务 `product-service`、物流微服务 `logistics-service`，以及 Agent 内部调用的导购工具集和售后 BPMN 流程。系统主要 API 接口包括 Web 页面入口 `/`、Prometheus 指标接口 `/metrics` 和对话接口 `/api/chat`。其中 `/api/chat` 会根据用户意图路由到订单、物流、商品咨询、售后 BPMN 或增强导购 Agent。

### 4.3.1 效率评价

效率主要通过响应时间评价。本次监控中，20 次请求的平均延迟为 23858.95 ms，P50 延迟为 22113.08 ms，P95 延迟为 48661.17 ms，最大延迟为 49792.27 ms。Prometheus 当前查询到的 P95 延迟约为 28.5 秒。

从结果看，系统能够稳定返回结果，但在接入真实 LLM 的导购链路下，响应时间明显较长。导购增强路径需要先收集商品候选、评价摘要、销量信号、店铺售后信息，再由真实 LLM 基于 evidence 证据包进行综合推荐推理，因此耗时高于普通规则查询和 BPMN 流程。

效率方面的结论是：普通服务和基础查询路径可以满足课程演示需求，真实 LLM 导购路径具备完整推理能力，但响应时间偏长，仍有优化空间。后续可以通过缓存商品 evidence、缓存导购结果、区分普通商品查询与购买决策、使用流式响应、设置模型调用超时与降级策略等方式降低用户等待时间。

### 4.3.2 可用性评价

可用性采用成功率进行评价：

```text
可用性 = 成功请求数 / 总请求数
```

本次监控中总请求数为 20，成功请求数为 20，失败请求数为 0，成功率为 100.00%。Prometheus 当前接口成功率查询结果也为 1.0，即 100%。Kubernetes 中各 Pod 均为 Running，Deployment 均为 Ready，说明服务在本次监控期间可用性较好。

可用性方面的结论是：系统在当前演示负载下能够稳定处理订单、物流、售后和导购请求；K8s 编排和 Docker Compose 本地部署均可支撑服务运行；Prometheus Targets 显示监控目标均为 UP，说明服务监控链路也处于可用状态。

### 4.3.3 健壮性评价

健壮性主要从异常处理、fallback 能力、容器运行稳定性和自动化测试保障四个方面评价。

系统具备以下健壮性设计：第一，若没有配置 `OPENAI_API_KEY`，`llm.py` 会自动使用 MockLLM；第二，导购 Agent 在真实 LLM 不可用、调用失败或 evidence 为空时，会回退到 `tools.recommend_products` 规则推荐；第三，Kubernetes Deployment 管理 Pod，服务异常时可自动重启；第四，Jenkins 在部署前执行 `py_compile` 和 `evaluate.py`，部署后执行 Smoke Test，降低错误版本进入运行环境的概率。

从本次监控结果看，20 次请求全部成功，失败请求数为 0；Docker Compose 中业务服务和监控服务均处于 Up 状态；Kubernetes Pod Restart 次数为 0。因此，本次健壮性评价为较好。需要注意的是，真实 LLM 请求耗时较高，未来生产环境还应增加更细粒度的重试、熔断、限流、异步处理和缓存策略。

### 4.3.4 吞吐率评价

吞吐率计算公式为：

```text
吞吐率 = 总请求数 / 总耗时
```

本次 `quality_monitor.py` 采集到的吞吐率为 0.079 req/s；Prometheus 基于最近 1 分钟窗口查询到的 HTTP 吞吐率约为 0.618 req/s。两者统计口径不同：脚本统计的是完整业务请求端到端耗时，包含真实 LLM 长耗时推理；Prometheus 统计的是窗口内 HTTP 请求速率，包含页面、指标和接口请求。

吞吐率方面的结论是：当前系统适合课程演示和功能验证，但在真实 LLM 参与推理时吞吐率偏低。如果面向生产环境，需要通过多副本扩容、异步任务、结果缓存、模型网关优化和更大规模压测来提升并发处理能力。

### 4.3.5 资源消耗评价

资源消耗方面，Web Agent 容器 `service-agent-lab-app-1` CPU 使用率约 0.07%，内存约 56.16 MiB；订单、商品、物流微服务内存约 13 MiB，CPU 使用率约 0.01%；Prometheus、Grafana、cAdvisor 三个监控容器内存分别约 64.38 MiB、87.04 MiB、114.3 MiB。整体来看，业务服务本身资源占用较低，监控组件资源占用也处于本地演示环境可接受范围内。

资源消耗方面的结论是：系统在课程实验规模下资源开销较小，Docker Desktop 本地环境可以同时运行业务服务、监控服务和 K8s 服务。若部署到生产环境，需要进一步设置容器 CPU/内存 requests 与 limits，并使用 Prometheus/Grafana 长期观测资源趋势。

### 4.3.6 综合评价

综合来看，本项目已经形成从开发、测试、构建、部署到监控评价的闭环。DevOps 方面，项目使用 GitHub 管理代码，使用 Jenkins 自动执行测试、构建镜像和部署，使用 Docker/Kubernetes 实现服务容器化和编排，使用 Kubernetes Secret 管理真实 LLM 配置。服务监控方面，项目接入 Prometheus、Grafana 和 cAdvisor，能够采集调用参数、时间参数和资源消耗参数，并通过 `quality_monitor.py` 产生业务流量和评价数据。

本次服务质量评价结果显示：系统可用性较好，20 次业务请求全部成功；健壮性较好，容器和 K8s Pod 均稳定运行；资源消耗较低，适合课程演示和本地部署；但效率和吞吐率仍受真实 LLM 调用耗时影响，存在优化空间。总体而言，当前系统满足服务工程实践中“可运行、可部署、可监控、可评价”的实验要求，适合作为 Agent 服务工程与 DevOps 综合实践项目。
