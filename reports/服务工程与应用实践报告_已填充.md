# 第 1 部分 服务创新设计

## 1. 用户需求调研

校园电商和外卖服务系统中，用户常见需求包括订单查询、物流状态查询、商品价格库存咨询、退换货政策咨询、售后退款申请以及购买决策辅助。传统客服系统通常能够回答“订单到哪了”“商品多少钱”“能不能退货”等确定性问题，但在“我想买一个 300 元以内的蓝牙耳机，宿舍和图书馆用，重视降噪和续航，不想漏音”这类购买决策问题上，普通商品查询能力不足。

用户在导购场景中的真实需求具有以下特点：

1. 需求表达自然化。用户往往用口语描述预算、场景和偏好，而不是给出结构化参数。
2. 决策因素多维化。导购不只看价格，还要综合评价、销量、店铺评分、售后评分、风险点等信息。
3. 推荐结果需要解释。用户需要知道为什么推荐某个商品，为什么不推荐另一个商品。
4. 结果需要稳定可靠。即使真实 LLM 不可用，系统也需要给出可演示、可解释的 fallback 推荐。

因此，本项目在保留原有客服系统的基础上，重点增强导购路径，使其具备“工具收集证据 + evidence 证据包 + LLM 推理推荐”的能力。

## 2. 服务蓝图设计

系统服务蓝图分为用户层、前端交互层、Agent 编排层、业务工具层、微服务层和数据层。

| 层次 | 内容 |
|---|---|
| 用户层 | 用户通过 Web 页面输入订单、物流、商品、售后、导购等问题 |
| 前端交互层 | `web/index.html` 展示对话内容、意图识别结果和 Agent trace |
| 服务入口层 | `server.py` 提供 HTTP 服务，`app.py` 执行护栏、编排和输出脱敏 |
| Agent 编排层 | `agent.py` 负责路由、专家分发、ReAct Agent 和导购专家 |
| 工作流层 | `bpmn_engine.py` 执行 `aftersale.bpmn` 售后退款流程 |
| 工具层 | `tools.py` 封装订单、物流、商品、政策检索、商品推荐等工具 |
| 导购增强层 | `shopping_research_agent.py` 构造导购工具集和 evidence 证据包 |
| 微服务层 | `services/order_service.py`、`product_service.py`、`logistics_service.py` |
| 数据层 | `data.py` 提供订单、商品、扩展商品信息和政策知识库模拟数据 |

总体服务流程为：

```text

```

## 4. 服务模式与创新设计分析

本系统的服务创新点是将导购服务从普通问答升级为证据驱动的个性化购买决策服务。

原系统主要解决确定性客服问题，如订单状态、物流信息、商品价格、售后政策等。增强后系统能够处理用户复杂购买需求，具体体现在：

1. LLM 需求理解：提取预算、商品类别、使用场景、偏好和避雷点。
2. 工具化证据收集：通过导购工具收集候选商品、评价摘要、销量信号、店铺和售后评分。
3. evidence 证据包：将分散商品数据结构化，提供给 LLM 作为推荐依据。
4. 证据约束推理：提示词要求 LLM 不编造 evidence 中没有的数据。
5. 风险解释能力：输出风险提醒和不推荐商品，而不仅仅输出 Top3。
6. fallback 稳定性：真实 LLM 不可用时，自动回退到规则推荐。

该设计体现了“工具增强 + 证据约束 + 大模型推理 + 可降级运行”的服务创新思路。

# 第 2 部分 业务流程建模

## 1. 业务流程建模设计

### 1.1 流程整体设计

系统整体业务流程由统一入口、路由分发和多业务路径组成。用户请求进入系统后，先经过输入护栏，防止提示注入和敏感请求。随后由路由 Agent 判断意图，分发到售后、物流、商品咨询、导购或其他路径。

当前项目包含以下主要路径：

| 业务类型 | 实现路径 |
|---|---|
| 售后退款 | `agent.router -> expert_aftersale -> run_aftersale -> aftersale.bpmn` |
| 物流查询 | `router -> expert_logistics -> react_agent -> query_order/track_logistics/search_policy` |
| 普通商品查询 | `router -> expert_shopping -> react_agent -> query_product` |
| 增强导购 | `router -> expert_shopping -> shopping_research_agent` |
| 其他问题 | 默认帮助答复 |

报告用完整流程图位于：

```text
flows/shopping_research_agent_report.bpmn
```

该 BPMN 文件仅用于报告展示，不接入运行路径。它合并展示了输入护栏、路由、售后 BPMN、物流查询、商品咨询、增强导购和输出脱敏等完整流程。

### 1.2 任务节点设计与实现

售后路径使用真实 BPMN 执行，核心节点如下：

| BPMN 节点 | 对应处理函数 |
|---|---|
| 查询订单 | `h_query_order` |
| 查超时补偿政策 | `h_compensation` |
| 查退款政策 | `h_refund_policy` |
| 人工审核 | `h_manual_review` |
| 自动发起退款 | `h_auto_refund` |
| 通知用户 | `h_notify` |

增强导购路径虽然不由 BPMN 执行，但在报告用 BPMN 中建模为如下节点：

| 节点 | 实现 |
|---|---|
| 需求理解 | `_understand_need`，真实 LLM 或规则抽取 |
| 检索候选商品 | `collect_product_candidates` |
| 收集评价摘要 | `collect_review_summary` |
| 收集销量信号 | `collect_sales_signal` |
| 收集店铺售后信息 | `collect_store_profile` |
| 构造 evidence | `shopping_research_agent` 内部 evidence 字典 |
| 真实 LLM 推理 | `_reason_with_llm` |
| fallback 推荐 | `_fallback -> recommend_products` |

### 1.3 微服务设计与实现

项目使用三个轻量微服务模拟真实业务服务：

| 微服务 | 文件 | 端口 | 功能 |
|---|---|---|---|
| 订单服务 | `services/order_service.py` | 8001 | 查询订单详情、发起退款 |
| 商品服务 | `services/product_service.py` | 8002 | 查询商品价格库存、搜索候选商品、查询商品详情 |
| 物流服务 | `services/logistics_service.py` | 8003 | 查询配送/物流状态和是否超时 |

工具层 `tools.py` 对这些微服务进行封装，使 LLM 不直接访问 HTTP 接口，而是通过工具契约调用：

```text
query_order
track_logistics
query_product
search_products
get_product_detail
recommend_products
refund_order
search_policy
```

### 1.4 Agent 设计与实现

Agent 设计分为三层：

1. 路由 Agent：判断用户意图为售后、物流、导购或其他。
2. 专家 Agent：根据不同业务进入物流专家、售后专家或导购专家。
3. ReAct Agent：在复杂查询中由 LLM 决定调用哪个工具。

核心代码位于 `agent.py`：

```python
EXPERTS = {
    "物流": expert_logistics,
    "售后": expert_aftersale,
    "导购": expert_shopping
}
```

增强导购专家逻辑为：

```text
如果是购买决策类问题 -> shopping_research_agent
如果是简单商品咨询 -> react_agent / query_product
```

这样保证新增导购能力不会破坏原有普通商品查询。

## 3. 工作流创新性分析及效果展示

传统 BPMN 更适合描述固定业务流程，例如售后退款审批；而导购场景具有较强的不确定性和推理性，需要结合用户偏好、商品评价、风险点、销量、店铺售后等多维证据。本项目将两类思想结合：

1. 售后退款仍使用 BPMN 流程，保证业务流程清晰、可控。
2. 导购推荐使用 Agent + 工具 + evidence + LLM 推理，增强个性化决策能力。
3. 报告用 BPMN 总流程图展示系统整体服务流程，但不强行让导购由 BPMN 驱动。

这种设计既保留了流程建模的规范性，又体现了大模型在复杂服务决策中的创新应用。

# 第 3 部分 系统设计与实现

## 1. 面向服务架构设计方案

系统采用面向服务架构，将业务能力拆分为前端、服务入口、Agent 编排、工具层、微服务层、数据层和工作流层。

系统架构如下：

```text
Web 前端
  -> server.py
  -> app.py
  -> guardrails.py
  -> agent.py
      -> react_agent
      -> expert_aftersale
      -> expert_shopping
  -> tools.py
      -> order_service
      -> product_service
      -> logistics_service
      -> rag.py
  -> bpmn_engine.py / bpmn_handlers.py
  -> shopping_research_agent.py
  -> llm.py
```

该架构具有以下特点：

1. 服务边界清晰。订单、商品和物流分别由独立服务模拟。
2. 工具契约统一。LLM 通过 `tools.py` 中定义的工具访问业务能力。
3. 工作流可建模。售后退款由 BPMN 文件定义和执行。
4. LLM 可替换。`llm.py` 支持 MockLLM 和真实 OpenAI-compatible API。
5. 导购可降级。真实 LLM 不可用时，规则推荐可作为 fallback。

## 2. 系统实现方案

### 2.1 项目结构

| 文件/目录 | 作用 |

### 2.3 增强导购 Agent 实现

新增 `shopping_research_agent.py`，核心函数为：

```python
def shopping_research_agent(text: str, user_id: str = "u001", verbose: bool = True) -> dict
```

返回结构：

```python
{
  "answer": "...",
  "trace": [...],
  "evidence": {...},
  "mode": "real_llm" 或 "fallback_rules"
}
```

导购工具集包括：

| 工具函数 | 功能 |
|---|---|
| `collect_product_candidates` | 检索候选商品 |
| `collect_review_summary` | 收集评价摘要和风险点 |
| `collect_sales_signal` | 收集价格、评分和月销量 |
| `collect_store_profile` | 收集店铺评分和售后评分 |

构造的 evidence 证据包包括：

```python
{
  "user_need": 原始需求,
  "parsed_need": 结构化需求,
  "candidates": 候选商品,
  "review_summary": 评价摘要,
  "sales_signal": 销量信号,
  "store_profile": 店铺售后信息,
  "risk_points": 风险点,
  "source_note": 数据来源说明
}
```

LLM 提示词要求：

1. 不编造 evidence 中没有的数据。
2. 如果数据来自 demo_data，要说明当前基于演示数据/样例数据。
3. 推荐结论必须对应用户预算、场景、偏好和避雷点。
4. 输出 Top3 推荐、推荐理由、风险提醒、不推荐商品、适合/不适合场景和数据来源说明。

## 4. 面向服务技术的创新应用

本项目的主要创新是将导购服务升级为工具增强型 LLM Agent。原系统已有订单、物流、商品、售后、RAG 和 BPMN 等功能，本次增强不是重写系统，而是在原导购路径上引入以下能力：

1. 真实 LLM 需求理解：从自然语言中抽取预算、场景、偏好和避雷点。
2. 导购工具集：通过固定编排调用四类工具收集商品证据。
3. evidence 证据包：把商品候选、评价、销量、店铺售后和风险点统一结构化。
4. 证据约束推理：真实 LLM 基于 evidence 输出推荐，避免凭空生成。
5. 风险感知推荐：不仅推荐商品，还说明不推荐项和风险提醒。
6. fallback 机制：真实 LLM 不可用时自动回退到规则推荐。

当前工具集是固定编排调用：

```text
collect_product_candidates
-> collect_review_summary
-> collect_sales_signal
-> collect_store_profile
-> evidence
-> LLM 推理
```

与生产级 Agent 的区别在于：真实业务中通常由 LLM 或 Planner 动态选择工具，并接入真实商品库、库存、价格、用户画像、优惠券、评论系统等多源数据。本项目采用固定工具链，是为了在教学实验中保证稳定、可解释和易展示。二者核心思想一致：LLM 不凭空回答，而是在工具提供的结构化证据基础上完成推理和决策。

## 4.1 DevOps 技术运用

本项目采用“小步迭代、持续验证、可回滚”的 DevOps 思路组织开发与交付。由于项目包含 Agent 编排、BPMN 流程、模拟微服务、Web 展示和 LLM 接入等多个部分，如果只依赖人工演示，容易在新增导购功能时破坏原有订单、物流、商品咨询和售后路径。因此，本项目将代码结构、自动化测试、容器化部署和运行配置统一纳入 DevOps 流程。

项目使用 Git 进行版本管理，按功能边界组织代码：

| 模块 | 文件/目录 | 作用 |
|---|---|---|
| Agent 编排层 | `agent.py` | 负责意图识别、专家路由、ReAct 调用和导购路径接入 |
| LLM 接入层 | `llm.py` | 统一真实 LLM 与 MockLLM 的调用接口 |
